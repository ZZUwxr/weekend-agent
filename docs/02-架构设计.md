## 分层架构说明

### 1. API 层 (`api/`)
- **职责**：HTTP 请求路由、参数校验、依赖注入
- **特点**：轻薄无业务逻辑，仅负责请求/响应转换
- **依赖注入**：通过 `deps.py` 创建 Service 实例，支持运行时切换数据后端

### 2. Service 层 (`services/`)
- **职责**：业务流程编排，协调 Agent 与 Repository
- **核心类**：`PlanService` 封装编排器和会话存储，提供 `preview_plan`、`get_plan`、`confirm_plan`、`handle_event`

### 3. Agent 层 (`agent/`)
- **职责**：LLM 驱动的智能决策编排
- **核心**：`Orchestrator` 按 8 步流水线执行，每步调用一个 Skill
- **LLM 抽象**：支持 OpenAI 客户端和 Mock 客户端，可无缝切换

### 4. Tools 层 (`tools/`)
- **职责**：封装外部操作（POI 查询、路线计算、预约、叫车等）
- **特点**：统一的 `ToolResult` 返回结构，支持 Mock 场景

### 5. Repository 层 (`repositories/`)
- **职责**：数据访问抽象
- **双后端**：支持 JSON 文件和 PostgreSQL，通过配置切换
- **回退机制**：优先读取完整数据文件，不存在时回退 `.sample.json`

### 6. Domain 层 (`domain/`)
- **职责**：领域模型、枚举、评分逻辑
- **特点**：Pydantic v2 模型贯穿所有层，作为统一数据载体

## 设计决策

### 决策1：Skill 流水线编排
- **背景**：周末规划涉及多个子任务（理解用户、选择地点、规划路线等）
- **选择**：8 个独立 Skill 串行编排，每步可发射 SSE 事件
- **原因**：便于调试和扩展，每个 Skill 可独立测试和替换

### 决策2：JSON / PostgreSQL 双数据后端
- **背景**：开发阶段需要快速迭代，生产需要持久化存储
- **选择**：通过 `DATA_BACKEND` 配置切换 json / postgres
- **原因**：JSON 后端零依赖即可运行，PostgreSQL 后端支持空间查询和并发

### 决策3：Mock LLM 降级
- **背景**：开发和测试阶段可能没有 LLM API Key
- **选择**：`LLM_PROVIDER=mock` 使用规则引擎生成结构化输出
- **原因**：保证服务在无 LLM 时仍可运行，便于端到端测试

### 决策4：SSE 流式输出
- **背景**：规划过程涉及多步 LLM 调用，耗时较长
- **选择**：通过 Server-Sent Events 实时推送每步进度
- **原因**：提升用户体验，前端可展示进度条和中间结果
