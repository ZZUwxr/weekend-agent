## 核心数据流

### 数据流1：规划预览（同步）

**触发条件**：用户提交规划请求 `POST /api/v1/plans/preview`

**流向**：
1. 前端发送 `PlanPreviewRequest`（包含自然语言描述、时间、位置等）
2. API 层校验请求参数，通过依赖注入获取 `PlanService`
3. `PlanService.preview_plan()` 调用 `Orchestrator.preview_plan()`
4. Orchestrator 按 8 步流水线执行：
   - ① `UserUnderstandingSkill`：调用 LLM 解析用户意图 → 生成 `GroupContext`
   - ② `ConflictDetectionSkill`：调用 LLM 识别冲突 → 生成 `Conflict[]`
   - ③ `NegotiationSkill`：调用 LLM 生成协商策略 → 生成 `NegotiationStrategy[]`
   - ④ `ExperiencePlanningSkill`：调用 LLM 规划体验骨架 → 生成 `PlanCandidate[]`
   - ⑤ `PlaceSelectionSkill`：从 Repository 查询 POI → 调用 LLM 选择主备 POI
   - ⑥ `RoutingSkill`：从 Repository 查询路线 → 规划转场
   - ⑦ `TimelineBuilderSkill`：调用 LLM 构建时间轴
   - ⑧ `scoring.py`：计算评分 → 选出推荐方案
5. 构建 `ExecutionTask` 图（预约、叫车、分享任务）
6. 将 `PlanOutput` 存入 `SessionStore`
7. 返回 `PlanOutput` 给前端

**涉及数据**：
```python
# 请求
PlanPreviewRequest(user_id, query, city, start_time, duration_minutes, location)

# 响应
PlanOutput(session_id, inferred_context, conflicts, plan_candidates, recommended_plan_id, ...)
```

### 数据流2：流式预览（SSE）

**触发条件**：用户提交流式规划请求 `POST /api/v1/plans/preview/stream`

**流向**：
1. API 层创建 `asyncio.Queue`，启动异步任务运行 `Orchestrator`
2. Orchestrator 每步执行前发射 `step_start` 事件
3. 每步执行中调用工具时发射 `tool_call` 事件
4. 每步执行完成后发射 `step_complete` 事件
5. 候选方案生成时发射 `candidate_start` / `candidate_complete` 事件
6. 全部完成后发射 `plan_complete` 事件
7. 前端通过 EventSource 实时接收事件

**SSE 事件流示例**：
```
event: step_start
data: {"step": "user_understanding"}

event: tool_call
data: {"tool": "llm", "input": "..."}

event: step_complete
data: {"step": "user_understanding", "result": {...}}

event: step_start
data: {"step": "conflict_detection"}

...

event: plan_complete
data: {"session_id": "sess_xxx", "recommended_plan_id": "plan_a"}
```

### 数据流3：事件驱动重规划

**触发条件**：外部事件（排队变更、天气变化等）`POST /api/v1/plans/{session_id}/events`

**流向**：
1. 前端发送 `PlanEvent`（事件类型、影响的 POI、严重度等）
2. `PlanService.handle_event()` 从 `SessionStore` 获取已有方案
3. `Orchestrator.replan()` 基于事件影响范围部分重规划
4. 生成新版本 `PlanOutput`（`plan_version` 递增）
5. 存入 `SessionStore`，返回新方案

### 数据流4：方案执行

**触发条件**：用户确认方案后执行 `POST /api/v1/plans/{session_id}/execute`

**流向**：
1. 从 `SessionStore` 获取已确认的方案
2. 遍历 `execution_graph` 中的 `ExecutionTask`
3. 按依赖顺序执行任务（预约、叫车、分享）
4. 每个任务调用对应的 Tool（`booking_tool`、`taxi_tool`、`share_tool`）
5. 返回执行结果

## 数据模型关系

```
PlanOutput (聚合根)
├── GroupContext
│   └── RoleProfile[]
├── Conflict[]
├── NegotiationStrategy[]
├── PlanCandidate[] (Plan A / Plan B / Recommended)
│   ├── Stage[]
│   │   └── POI (selected + fallback)
│   ├── TimelineItem[]
│   ├── SatisfactionScore[]
│   └── route_segments[]
├── ExecutionTask[]
└── PlanEvent[] (触发重规划)
```

## 状态管理

### 方案状态流转

```
PENDING ──(preview)──▶ PREVIEW ──(confirm)──▶ CONFIRMED ──(execute)──▶ EXECUTED
                         │
                         ├──(event)──▶ REPLANNING ──▶ PREVIEW (plan_version++)
                         │
                         └──(timeout)──▶ EXPIRED
```

### 会话存储

- 存储方式：内存字典（`SessionStore`）
- Key：`session_id`
- Value：`PlanOutput`
- 生命周期：服务运行期间有效，重启后清除

## 数据源

### JSON 文件数据（`app/data/`）

| 文件 | 说明 | 记录数 |
|------|------|--------|
| `poi.json` | POI 主数据 | ~150 |
| `route_edges.json` | POI 间路由边 | ~200 |
| `queue_status.json` | 排队状态 | ~150 |
| `weather.sample.json` | 天气样本 | ~7 |
| `user_profiles.sample.json` | 用户画像样本 | ~10 |
| `booking_records.sample.json` | 预约记录样本 | ~20 |

### PostgreSQL 数据（可选）

通过 `DATA_BACKEND=postgres` 切换，使用 `database_url` 连接 PostgreSQL + PostGIS。
